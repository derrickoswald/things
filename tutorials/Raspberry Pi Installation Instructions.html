Raspberry Pi
Installation Instructions

Overview

These instructions cover installing the things system (https://github.com/derrickoswald/things) on a Raspberry Pi single board computer (https://www.raspberrypi.org/). Although this was tested with the original board, the instructions should be the same for a Raspberry Pi 2 board.

Prerequisites

Install Raspbian on an Sdcard bigger than 2GB following instructions on http://raspbian.org/. Extend the file system to use the entire SD card and set the locale and time zone, e.g. en_US.UTF-8 UTF-8, using raspi-config as described at https://www.raspberrypi.org/documentation/configuration/raspi-config.md.


CouchDB

Installation

Check Version

Establish an ssh terminal session to your Pi using the IP address, the default password is raspberry:

ssh pi@<pi IP address> e.g. ssh pi@192.168.10.142

pi@192.168.10.142's password: raspberry

You should check the version of CouchDB that would be installed by default using apt-get (i.e. the version that would be installed by using command sudo apt-get install couchdb), but at the time of writing, the version was only 1.2 and the things system relies on version 1.6

pi@raspberrypi ~ $ aptitude show couchdb

Package: couchdb

State: not installed

Version: 1.2.0-5

Custom Install

Instead follow the instructions that are found at https://cwiki.apache.org/confluence/display/COUCHDB/Debian for Debian versions >= 7.0, and CouchDB up to 1.6.x, except for the following changes:

(see also http://jeeonthepi.blogspot.ch/2014/08/installing-couchdb-1.html)

  • install libmozjs185-1.0 instead of libmozjs185

  • skip the install of build-essentials, it's already the most recent

  • change the ownership of /usr/local/etc/couchdb/local.ini to couchdb

So the sequence of commands is:

sudo apt-get install -y lsb-release

echo "deb http://binaries.erlang-solutions.com/debian `lsb_release -cs` contrib" | sudo tee /etc/apt/sources.list.d/erlang-solutions.list

wget -O - http://binaries.erlang-solutions.com/debian/erlang_solutions.asc | sudo apt-key add -

sudo apt-get update -y


sudo apt-get install -y libmozjs185-1.0

sudo apt-get install -y libmozjs185-dev


sudo apt-get install -y erlang-nox erlang-dev


sudo apt-get install -y libnspr4 libnspr4-0d libnspr4-dev libcurl4-openssl-dev curl libicu-dev


sudo useradd -d /var/lib/couchdb couchdb

sudo mkdir -p /usr/local/{lib,etc}/couchdb /usr/local/var/{lib,log,run}/couchdb /var/lib/couchdb

sudo chown -R couchdb:couchdb /usr/local/{lib,etc}/couchdb /usr/local/var/{lib,log,run}/couchdb

sudo chmod -R g+rw /usr/local/{lib,etc}/couchdb /usr/local/var/{lib,log,run}/couchdb


wget http://mirror.switch.ch/mirror/apache/dist/couchdb/source/1.6.1/apache-couchdb-1.6.1.tar.gz

tar xzf apache-couchdb-1.6.1.tar.gz

apache-couchdb-1.6.1.tar.gz

cd apache-couchdb-1.6.1/


./configure --prefix=/usr/local --with-js-lib=/usr/lib --with-js-include=/usr/include/mozjs --enable-init

make

sudo make install


sudo ln -s /usr/local/etc/init.d/couchdb /etc/init.d/couchdb

sudo /etc/init.d/couchdb start

sudo update-rc.d couchdb defaults

sudo chown couchdb:couchdb /usr/local/etc/couchdb/local.ini

Notes

Configuration Files

In many of the following steps you are instructed to edit the CouchDB configuration files. You can locate the CouchDB configuration files using the -c option:

couchdb -c

/usr/local/etc/couchdb/default.ini

/usr/local/etc/couchdb/local.ini

By design, you edit the local.ini file because this is maintained across upgrades, whereas the default.ini file is overwritten when a new upgrade installation is performed. The values in the local.ini file supersede the defaults, since that file is listed last. You must be root or the couchdb user to edit these files, so sudo is normally required to edit them.

Log Files

Is some of the steps listed below you are instructed to monitor the CouchDB log. The location of the log file can be found by checking the log/file entry in the configuration.

vi /usr/local/etc/couchdb/default.ini

…

[log]

file = /usr/local/var/log/couchdb/couch.log

level = info

…

You can then monitor the log with the tail command, with the optional -f switch to continuously monitor:

sudo tail -f /usr/local/var/log/couchdb/couch.log

HTTP

In this step you will change the bind address of the CouchDB service to expose the HTTP service. Edit the local configuration file and change the bind_address in the httpd section to 0.0.0.0 (choose any) or 127.0.0.1, the localhost address, and then restart CouchDB:

sudo vi /usr/local/etc/couchdb/local.ini

…

[httpd]

bind_address = 0.0.0.0

…

sudo service couchdb restart

Test

You can test if it works by fetching the root page from the CouchDB server:

curl http://127.0.0.1:5984/

Which should respond with something like (your version and uuid may differ):

{"couchdb":"Welcome","uuid":"2bfcb62ac1b0338cac0c993076d240a9","version":"1.6.1","vendor":{"name":"The Apache Software Foundation","version":"1.6.1"}}

Query Servers

There is a problem in storing large JSON files (actually indexing them is the problem). After hunting around this fix seemed to work. Edit the couchdb local.ini, and add the javascript query_server with the -S parameter to specify using up to 128 Mbytes:

sudo vi /usr/local/etc/couchdb/local.ini

…

[query_servers]

javascript = /usr/local/bin/couchjs -S 134217728 /usr/local/share/couchdb/server/main.js

…

Then restart CouchDB to re-read the initialization files:

sudo service couchdb restart

Security

When CouchDB is installed, it has no security enabled – all visitors can do all things – which is called Admin Party. In this step you use Futon to turn off the Admin Party mode by creating an admin user.

Browse to http://<pi IP address>:5984/_utils/, e.g. http://192.168.10.142:5984/_utils/. This should display the Futon user interface <TBD>, the new version of CouchDB uses the Fauxton UI</TBD>.

Click the “Fix this” link in the lower right hand corner of Futon's main page. Provide a user name and a password. This document and others refers to the admin user when referring to the name chosen here.

Test

You can check if security is enabled by browsing to the Futon Overview page, e.g. http://<pi IP address>:5984/_utils/ and click on the Create Database link in the upper left corner. Enter a Database Name field in the dialog box and click the Create button. A message like You are not a server admin. should be displayed indicating it wasn't successful.

Only if you first login by clicking the Login link in the lower right will you be able to create a database or do other administrative functions. (you can delete your test database by clicking into the database link from the list on the Overview page and clicking the Delete Database link).

NOTE: The login validity will time out after a default 10 minutes (100 seconds in couch_httpd_auth/timeout in .ini file).

This is a bootstrap procedure to get an initial installation up and running- subsequently you can use the program itself to do this replication. <TBD>An alternative is to copy a things database data file containing the design document into the data file location of CouchDB.</TBD>

In the form, click the Remote database radio button under Replicate changes from:, and enter the remote host things database URL, e.g. http://thingtracker.no-ip.org/root/things or your development machine instance, e.g. http://192.168.10.103:5984/things. Under to:, enter “things” as the local database name.


Push the Replicate button. Click OK in the This will create a database named things. Ok? confirmation dialog.

Deluge

Installation

In this step the Deluge BitTorrent client is installed and made available to the browser client via the CouchDB proxy mechanism. In this case, the version installed by the debian package manager is sufficient. So the steps are:

sudo apt-get install deluge

sudo apt-get install deluged

sudo apt-get install deluge-web

Test

To test the instalation, you can manually start the web component:

deluge-web

and navigate to the web page http://<pi IP address>:8112/

The default password is: deluge

Autostart

To set up the pi so that when it boots the deluge web services are running, you can follow the autostart instructions ar http://www.onepitwopi.com/raspberry-pi/installing-deluge-1-3-6-on-a-raspberry-pi/. Start by adding the pi user to the authorization list for deluge:


vi ~/.config/deluge/auth

add:

…

pi:raspberry:10

…

Then add the pi user as the user to run the deluge daemon as:

sudo vi /etc/default/deluge-daemon

…

# Configuration for /etc/init.d/deluge-daemon

# The init.d script will only run if this variable non-empty.

DELUGED_USER="pi"

# Should we run at startup?

RUN_AT_STARTUP="YES"

…

The init.d script, available at to start the deluge daemon and web service needs to be edited to change the location of the two binaries, DAEMON1 and DAEMON2:

sudo vi /etc/init.d/deluge-daemon

…

DESC="Deluge Daemon"

NAME1="deluged"

NAME2="deluge"

DAEMON1=/usr/bin/deluged

DAEMON1_ARGS="-d" # Consult `man deluged` for more options

DAEMON2=/usr/bin/deluge-web

DAEMON2_ARGS="" # Consult `man deluge-web` for more options

…

Then make the script executable:

sudo chmod +x /etc/init.d/deluge-daemon

There is still a small problem, in that the web service doesn't automatically connect to the deluged daemon. That is, navigating to the web service on port 8112 works, but the first thing you must do is connect to the deluged daemon.

To get deluge-web to connect to the daemon automatically, stop the daemon, then fill in the default_daemon value and make the file read-only:

sudo vi ~/.config/deluge/web.conf

…

"session_timeout": 3600,

"default_daemon": "127.0.0.1:58846",

"sidebar_multiple_filters": true

…

chmod -w ~/.config/deluge/web.conf

sudo reboot

After much trial and error in getting node set up on the pi, the solution that seems to work best is published by Adafruit (https://learn.adafruit.com/node-embedded-development/installing-node-dot-js) in the Raspberry Pi Package Repository:

The first step is to add the repository to the available sources list:

Then, installing node is as simple as

Test

To test the installation, logout and log back in and check that the node command works and you hava a recent version with:

node --version

v0.12.0

Nano

There are many ways to access CouchDB from Node, and after a cursory evaluation the nano package was selected. To install nano you can use the node package manager (npm) as follows:

sudo npm install nano -g

Path

To allow starting node processes from within CouchDB via the externals API, set the path to node modules using the NODE_PATH environment variable in the CouchDB startup:

sudo vi /usr/bin/couchdb

or

sudo vi /usr/local/bin/couchdb

and add the line:

export NODE_PATH=/usr/local/lib/node_modules

Or, another way to do the same thing without altering the environment (but only available in linux) is to link the installed global nano module in the same location you will install the server side script(s):

cd /home/pi

npm link nano

user_manager

Installation

NOTE: This describes the manual steps to set up the user_manager. The Configurator Wizard provides a much easier way to accomplish this.

The user_manager feature allows a non-logged in user to create their own account on a things system. The mechanism chosen to do this is a separate daemon started by CouchDB, that connects to CouchDB as an administrator, and that is accessed via proxy from a browser. In addition, the user_manager uses proxy authentication which has an additional two items that need to be set up, namely enabling proxy authentication and setting the shared secret.

First, download the user_manager script from GitHub to an approprate location (the command below assumes the home directory of the pi user):

cd /home/pi

wget https://raw.githubusercontent.com/derrickoswald/things/master/server/js/user_manager.js

Next, get a unique UUID to be used as the shared secret from CouchDB using the _uuids REST service:

curl http://127.0.0.1:5984/_uuids

This should respond with a JSON structure that contains a new UUID as a 32 character hexadecimal number:

{"uuids":["91c3b65ffa32d0478c33cd1f860016c5"]}

Edit the local configuration file for CouchDB:

sudo vi /usr/local/etc/couchdb/local.ini

or

sudo vi /etc/couchdb/local.ini

[Which one of the above to use can be determined with the command couchdb -c]

Add the proxy server for user_management under the httpd_global_handlers section:

[httpd_global_handlers]

…

user_manager = {couch_httpd_proxy, handle_proxy_req, <<"http://127.0.0.1:8000">>}

…

The port number must agree with the one chosen below.

Under the os_daemons section, add the command needed to start up the user_manager node process when CouchDB is started or restarted:

[os_daemons]

…

user_manager = /usr/local/bin/node /home/pi/user_manager.js

…

The path to node must match where node is installed (the path can be determined using the command which node) and the parameter must match where you placed the user_manager.js file.

Under the couch_httpd_auth section add the UUID determined above as the secret and set the proxy_use_secret value to true:

…

[couch_httpd_auth]

secret = 91c3b65ffa32d0478c33cd1f860016c5

proxy_use_secret = true

…


Lastly, under a custom properties section (that you need to create) called user_manager add the parameters for the program:

[user_manager]

listen_port = 8000

couchdb = http://127.0.0.1:5984/

username = admin

userrole = _admin

where the listen_port (8000) agrees with the httpd_global_handlers setting, the couchddb URL is what the user_manager program needs to connect to CouchDB locally on the host, and the username and userrole are the name and role for administrative access sufficient to create new users.

Restart couchdb:

sudo service couchdb restart

The CouchDB log should contain something like:

[Mon, 03 Aug 2015 15:23:35 GMT] [info] [<0.96.0>] Daemon "user_manager" :: listening on port 8000 as user admin/****


NOTE: To restart the user_manager service, just kill the one that's running and CouchDB will restart the process:

ps -ef | grep node

couchdb 15502 15479 4 17:25 ? 00:00:10 /usr/local/bin/node /home/pi/user_manager.js

sudo kill -9 15502

couchdb-lucene

The search integration chosen, uses the Apache Lucene engine coupled with the couchdb-lucene interface.

Since no pre-built binaries are available, the couchdb-lucene project must be built from scratch. As an overview, this involves installing a full Java JDK, not just a JRE, (to have access to the javac compiler) and Maven which is the build system, and then cloning and building the couchdb-lucene project from GitHub.

The proxy to the lucene-couchdb process needs to be added in the httpd_global_handlers section of the local configuration file::


[httpd_global_handlers]

…

_fti = {couch_httpd_proxy, handle_proxy_req, <<"http://127.0.0.1:5985">>}

…

The startup command for the lucene-couchdb process needs to e added to the os_daemons section:


[os_daemons]

…

couchdb_lucene = /home/pi/couchdb-lucene/target/couchdb-lucene-1.1.0-SNAPSHOT/bin/run

…

The startup script ( /home/pi/couchdb-lucene/target/couchdb-lucene-1.1.0-SNAPSHOT/bin/run) will also need the -server parameter removed from the java parameters to avoid the error:

Error occurred during initialization of VM

Server VM is only supported on ARMv7+ VFP

The CouchDB service should be restarted to recognize the changes:

sudo service couchdb restart

Test

Assuming that you have added some things, you can test the search capability with queries directly on the Lucene server like so (note that the port is 5985, one more than the CoucDB default):

curl http://localhost:5985/local/public_things/_design/public_things/search?q=bicycle

or via the proxy through CouchDB (see above):

curl http://localhost:5984/_fti/local/public_things/_design/public_things/search?q=bicycle

or when a vhost is active:

curl http://things.org/root/_fti/local/public_things/_design/public_things/search?q=bicycle

Or via POST with data like:

curl --data 'q=default:"smart phone"' http://localhost:5984/_fti/local/public_things/_design/public_things/search

Virtual Host

The normal URL used by CouchDB is somewhat long and counterintuitive. You can make a more user-friendly URL by using the vhost[link] capability of CouchDBpurposes if you have a suitable DNS entry. In this step, one experience in setting this up is described, but each installation will be different. For the purposes of the explanation, assume you have a DNS C record for things.org.

Router

It is likely that your Rasperry Pi is not connected directly to the internet, but is probably located on a subnet in the DMZ. Set up your edge router so that the DNS name hits the raspberry pi. For this example the router does a port translation as well from port 80 (the standard HTTP protocol port) to port 5984 (the standard CouchDB port):



vhost

Add a vhost entry in the local CouchDB initialization file that redirects HTTP requests at the root down to the _rewrite level (from where they will be rewitten according to CouchDB rewrite rules), and set the secure_rewrites setting to false:


sudo vi /usr/local/etc/couchdb/local.ini

…

[vhosts]

things.org = /things/_design/things/_rewrite/


[httpd]

secure_rewrites=false

…

Then you can restart CouchDB:

sudo service couchdb restart


Part of the installation of the things design document adds a set of rewrite rules. For example, there are already entries in the rewrites section like so:


{ "from": "root", "to": "../../.." },

{ "from": "root/*", "to": "../../../*" }


so that URLs that start with root are redirected up to the root URL. Within the program, at startup time, if the document URL does not contain the string “_rewrite” which indicates a vhost setting is actoive, a root prefix is added to all the CouchDB jQuery plugin calls by setting the prefix, like so:


$.couch.urlPrefix = “/root”;


in main.js. This setting coupled with the rewrite rules means the CouchDB server interface [link] e.g _all_dbs, _config etc. is available.

Configuration

After the components are set up, the things program itself can be used to update and maintain the CouchDB configuration. Internally it uses the _config server interface so that you don't have to. Navigate to the things system http://<pi IP address>:5984/things/_design/things/_rewrite/, e.g. http://192.168.10.142:5984/things/_design/things/_rewrite/, login as the administrator if necessary and then click the configuration tab and follow the instructions on screen.
